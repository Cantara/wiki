# Jira usage principles

Developers, project managers, testers and system administrators all have different experience and opinions with regards to how issue trackers should be used. Following the DevOps mindset, it makes sense to try to bridge our differences and try to agree on some common principles. The principles should work for Kanban and Scrum teams using Jira. 

1. Limit the number of issues which is _work in progress_ as this improves efficiency. See [Why Limit Work in Progress?](http://www.personalkanban.com/pk/featured/why<sub>~limit</sub><sub>work</sub><sub>in</sub><sub>progress/)Â and [Importance of Kanban work</sub><sub>in</sub><sub>progress (WIP) limits](http://www.atlanticcanadabusinessblog.com/index.php/2011/01/16/project</sub><sub>management/importance</sub><sub>of</sub><sub>kanban</sub><sub>work</sub><sub>in</sub><sub>progress</sub><sub>wip</sub>~limits/). 
1. Don't allocate issues upfront - let team member _pull_ an issue when she is ready to start on the task. This avoids the problem of a team member "locking tasks" which another team member could possibly start working on. Again, improves efficiency.  
1. Follow the default [Issue workflow](Issue-workflow.md). This is a basic, very generic flow which suites most agile development teams. Kanban teams can also use this scheme without any modifications.
